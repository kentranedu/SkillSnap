SkillSnap Capstone Project Documentation
Date: 2026-03-05

1) Project Summary
SkillSnap is a full-stack portfolio management app built with ASP.NET Core (API) and Blazor WebAssembly (client).
It lets users register/login with JWT authentication and manage portfolio project data through API-backed UI components.
The client presents project cards, tags, and API-fed lists with responsive layout improvements for desktop and mobile.

2) Key Features

A) CRUD Features
- Projects API supports:
  - Create: POST /api/projects (authenticated)
  - Read list: GET /api/projects (cached)
  - Read by id: GET /api/projects/{id}
  - Update: PUT /api/projects/{id} (authenticated)
  - Delete: DELETE /api/projects/{id} (authenticated)
- Skills API currently supports:
  - Read list: GET /api/skills
  - Create: POST /api/skills (Admin role only)

B) Security Features
- JWT-based authentication with login/register endpoints.
- Token validation enabled for issuer, audience, lifetime, and signing key.
- Protected project write endpoints using [Authorize].
- Role-based protection for skill creation using [Authorize(Roles = "Admin")].
- Client auth state stored in localStorage and attached as Bearer token to API requests.

C) Caching Features
- Projects list endpoint uses IMemoryCache with:
  - Fresh cache key (shorter TTL)
  - Fallback cache key (longer TTL)
- GET /api/projects returns X-Cache response header values: HIT, MISS, FALLBACK.
- After project create/update/delete, cache is invalidated and refreshed from DB to avoid stale reads.
- API logs include cache lifecycle events (invalidate/refresh) for verification.

D) Client State Features
- UserSessionService stores lightweight user context from JWT (user id, role, draft metadata).
- ProjectList keeps editing draft state and reloads API data after successful create.
- Home page composes reusable components (ProjectList, ProjectTags, ApiItemList) in a responsive grid.

3) Development Process and Use of Copilot
- Development followed an iterative workflow:
  1. Inspect existing API/client structure and auth configuration.
  2. Run live endpoint validation for auth, authorization, and CRUD behavior.
  3. Implement missing project CRUD endpoints and secure writes.
  4. Add cache refresh logic and log signals to verify data freshness.
  5. Refactor duplicate client service code into shared helper methods.
  6. Clean up unused code/services and improve naming consistency.
  7. Refresh UI spacing, typography, and responsive behavior.
  8. Rebuild and smoke-test after each major change.
- Copilot was used to:
  - Analyze controllers/services for security and logic gaps.
  - Generate and apply focused code patches.
  - Propose and implement refactors (helper extraction, naming cleanup).
  - Run command-line validation (build, runtime checks, endpoint tests).
  - Provide structured summaries of findings and implementation status.

4) Known Issues and Future Improvements

Known Issues
- Seed endpoint is currently mutable and should be restricted (auth/dev-only) before production use.
- Skills API does not yet include update/delete endpoints.
- Newly registered users are not automatically assigned Admin role, so skill creation is typically forbidden for them.
- JWT signing key in appsettings files is placeholder text and should be replaced with secure secret management.

Future Improvements
- Add full Skills CRUD (with clear ownership/role authorization policies).
- Apply secure-by-default authorization policy (fallback policy requiring auth, explicit [AllowAnonymous] where needed).
- Add lockout/rate-limiting protections for login to reduce brute-force risk.
- Add automated tests:
  - API integration tests for auth and CRUD authorization
  - Cache behavior tests for freshness after writes
  - UI component tests for empty/loading/error states
- Add user-facing profile and skill management pages.
- Add pagination/filter/sort for larger project lists.
- Add production-grade observability (structured logs, tracing, health checks).

5) Technical Q&A

Q1: Discuss development challenges.
- One challenge was endpoint completeness drift: client components expected project mutation endpoints while the API initially exposed only project reads. This required adding missing CRUD endpoints and aligning route behavior/status codes.
- Another challenge was balancing caching with data correctness. Read performance improved with IMemoryCache, but write operations initially risked stale responses unless cache invalidation/refresh was done immediately after CRUD operations.
- Security review surfaced a practical challenge: protection was attribute-based per endpoint, which can lead to accidental public mutation routes if [Authorize] is missed.
- UI challenges included improving readability and layout consistency across small and large screens while keeping existing component structure.

Q2: How did you structure business logic, data persistence, and state management?
- Business logic:
  - API controllers handle request validation, authorization checks, and operation orchestration.
  - Project cache lifecycle logic (hit/miss/fallback, invalidate/refresh) is centralized in ProjectsController helper methods.
  - Client services (AuthService, ProjectService) encapsulate HTTP communication and error handling.
- Data persistence:
  - Entity Framework Core with SQLite is used via SkillSnapContext.
  - Domain entities include PortfolioUser, Project, Skill, and Identity entities through ApplicationUser.
  - Controllers perform async EF operations with AsNoTracking for read-heavy paths.
- State management:
  - Auth token is persisted in localStorage and applied to HttpClient authorization headers.
  - UserSessionService stores lightweight user/session context (user id, role, draft metadata).
  - UI components refresh server-backed state after mutations to avoid stale client-side projections.

Q3: How did you implement security?
- JWT authentication was configured in ASP.NET Core with issuer, audience, lifetime, and signing key validation.
- Role/endpoint protection:
  - Project write operations require authenticated users.
  - Skill creation requires Admin role.
- Client security integration:
  - Token initialization and header propagation are handled centrally in AuthService.
  - Login/register error responses are normalized for reliable UI handling.
- Security validation was done through live endpoint tests for authorized and unauthorized scenarios.

Q4: What performance improvements did you apply?
- Added in-memory caching for project list reads with separate fresh/fallback cache keys and TTLs.
- Returned cache telemetry in response headers (X-Cache) to verify hit/miss behavior quickly.
- Implemented post-write cache invalidation and immediate rehydration from DB to maintain both speed and correctness.
- Added API logging for cache events (HIT/MISS/FALLBACK plus INVALIDATE/REFRESH) to support performance diagnostics.
- Used AsNoTracking on read paths to reduce EF Core tracking overhead for list retrieval endpoints.
